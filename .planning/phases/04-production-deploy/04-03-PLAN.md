  <name>Task 1: Verify production build meets performance budgets</name>
  <files></files>
  <read_first>vite.config.ts, package.json, vercel.json</read_first>
  <action>
  Run the production build and verify all performance budgets are met. This is a verification-only task -- no files are created or modified.

  **1. Run production build:**

  ```bash
  npm run build
  ```

  **2. Verify JS bundle size:**

  Examine the build output. Sum up all JS chunks that load on the initial page visit (entry chunk + vendor chunks). Lazy chunks (model-viewer, individual route pages) are NOT counted toward the initial budget.

  The target: total eagerly-loaded JS under 500 KB gzipped.

  If the build output does not clearly separate eager vs lazy chunks, check which chunks are referenced in dist/index.html -- those are the eager ones.

  **3. Verify built files exist:**

  ```bash
  ls -la dist/
  ls -la dist/assets/
  ```

  Confirm dist/index.html exists and references JS/CSS chunks. Confirm multiple JS chunks exist (evidence of code splitting).

  **4. Verify build has no TypeScript errors:**

  The `npm run build` script runs `tsc -b && vite build`. If it succeeds, TypeScript compilation passed.

  **5. If bundle is STILL over 500 KB gzipped:**

  Check if three.js is properly separated into its own chunk. If the initial chunk still contains three.js + R3F + all app code, investigate the manualChunks config. The three.js vendor chunk loads eagerly but caches separately. The key metric is: does model-viewer load lazily (not in the initial chunks)?

  Acceptable outcome: Eagerly-loaded JS (entry + vendor-three + vendor-react) under 500 KB gzipped total. If the sum is 480-500 KB, that is acceptable. If over 500 KB, additional splitting may be needed (e.g., separating @react-three/* into its own chunk).

  **6. Quick load time estimate:**

  500 KB over 4G (~1.5 MB/s effective throughput) = ~0.33 seconds for JS. Combined with HTML, CSS, no model loading on gallery page = well under 3 seconds. The 3-second budget is for initial page load, not 3D model loading.
